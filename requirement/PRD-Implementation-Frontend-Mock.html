🎯Problem Statement

ทีมเล็ก (2–15 คน) จัดการงานกระจัดกระจายในแชท สเปรดชีต และอีเมล ไม่มีจุดเดียวที่เห็นสถานะงานทั้งหมด ไม่รู้ว่าใครรับผิดชอบอะไร และงานตกหล่น ผู้ใช้ต้องการกระดาน Kanban ที่เห็นงานทั้งทีมในหน้าจอเดียว ลากการ์ดข้ามสถานะได้ลื่นไหล และเชื่อถือได้ว่าข้อมูลไม่หาย

ในเชิงการพัฒนา ทีมยังไม่ต้องการลงทุนกับโครงสร้าง backend (Node + PostgreSQL + auth จริง) ก่อนจะได้พิสูจน์ว่า core experience ของกระดาน — การสร้าง/ลาก-ย้าย card, ordering, optimistic UI — ใช้งานได้จริงและน่าใช้

💡Solution

สร้าง Kanban Board frontend เป็น React SPA ที่กดเล่นได้เต็มรูปแบบในเบราว์เซอร์ทันที โดยมี mock backend ที่เก็บ canonical state ไว้ใน localStorage และจำลองพฤติกรรมของ server จริง (async, latency, การ enforce สิทธิ์, การ broadcast การเปลี่ยนแปลงข้ามแท็บ)

ผู้ใช้เปิดแอปแล้วเลือกตัวตนจาก user-switcher (ตัวตนที่ seed ไว้) เห็นรายการ Board ที่ตนเป็น owner หรือ member เปิด Board เห็น Column เรียงแนวนอน แต่ละ Column มี Card ที่ลาก-ย้าย/จัดลำดับได้ด้วย drag-and-drop การลากตอบสนองทันที (optimistic) และคงอยู่หลัง refresh เปิดรายละเอียด Card ใน side-panel เพื่อแก้ description, assignee, due date, labels ได้

mock backend ถูกออกแบบให้ interface ของ API client ตรงกับ REST contract ของ PRD ต้นฉบับ (§3.7) เพื่อให้สลับไปต่อ Node backend จริงในอนาคตได้โดยแก้แค่ไส้ในของ client ชั้นเดียว ไม่ต้องแตะ UI, state store หรือ data model

📝User Stories — 42 ข้อ, 8 หมวด

Identity & Session 3

  1. 1As a user, I want to pick which seeded identity I am acting as from a switcher, so that I can simulate different team members without a real login.
  2. 2As a user, I want my current identity to persist across page refresh, so that I don't have to re-select it every time.
  3. 3As a user, I want to switch identity at any time, so that I can demonstrate collaboration between members on one machine.

Board list 6

  1. 4As a user, I want to see a list of all Boards where I am owner or member, so that I can switch between projects.
  2. 5As a user, I want Boards I have no access to to be hidden from my list, so that I only see what is relevant to me.
  3. 6As a user, I want to create a new Board with a name, so that I can organize work for a specific project.
  4. 7As a Board owner, I want to rename my Board, so that I can keep the name accurate.
  5. 8As a Board owner, I want to delete my Board, so that I can remove finished projects.
  6. 9As a member who is not the owner, I want the delete action to be unavailable, so that I cannot accidentally remove a Board I don't own.

Membership & invitation 4

  1. 10As a Board owner, I want to invite an existing seeded user by email, so that they can collaborate on the Board.
  2. 11As a Board owner, I want to remove a member, so that I can manage who has access.
  3. 12As an invited member, I want the Board to appear in my list when I switch to my identity, so that I can start working on it.
  4. 13As a user, I want to see the avatars of all Board members in the top bar, so that I know who is on the team.

Columns 6

  1. 14As a member, I want to create Columns with custom names, so that I can model my workflow stages.
  2. 15As a member, I want to rename a Column, so that I can adjust the workflow wording.
  3. 16As a member, I want to delete a Column, with confirmation when it still has Cards, so that I don't lose Cards by accident.
  4. 17As a member, I want to reorder Columns by dragging, so that the Board reflects the real sequence of work.
  5. 18As a member, I want each Column header to show a count of its Cards, so that I can see workload at a glance.
  6. 19As a member, I want an empty Board to show an inviting empty state to add the first Column, so that I know how to start.

Cards 5

  1. 20As a member, I want to create a Card with a title inside a Column, so that I can capture a task quickly.
  2. 21As a member, I want to open a Card in a side-panel to see and edit its full detail, so that the Board stays visible behind it.
  3. 22As a member, I want to edit a Card's description, so that I can capture full task detail.
  4. 23As a member, I want to delete a Card, so that I can remove tasks that are no longer relevant.
  5. 24As a member, I want an empty Column to show an inviting empty state to add the first Card, so that I know how to start.

Drag-and-drop & ordering 7

  1. 25As a member, I want to drag a Card to another Column, so that I can update its status visually.
  2. 26As a member, I want to reorder a Card within its Column, so that I can express priority.
  3. 27As a member, I want the moved Card to appear in its new place instantly (optimistic), so that the interaction feels immediate.
  4. 28As a member, I want a clear placeholder/ghost showing where the Card will drop, so that I can aim the drop accurately.
  5. 29As a member, I want a moved Card to stay where I dropped it after refresh, so that I trust the change was saved.
  6. 30As a member, I want a Card to snap back with an error message if the save fails, so that I am never misled about the saved state.
  7. 31As a member, I want to drag Cards and Columns using the keyboard, so that the Board is accessible.

Collaboration fields 6

  1. 32As a member, I want to assign a Card to a single teammate, so that responsibility is clear.
  2. 33As a member, I want to create colored Labels at the Board level, so that I can define categories like bug/feature/urgent.
  3. 34As a member, I want to attach and detach Labels on a Card, so that I can categorize it.
  4. 35As a member, I want a Card to show its Label color bar, assignee avatar, and due date for quick scanning, so that I don't need to open it.
  5. 36As a member, I want to set a due date on a Card, so that deadlines are explicit.
  6. 37As a member, I want overdue Cards highlighted with color plus an icon/text (not color alone), so that the cue is accessible to color-blind users.

Sync & reconcile 4

  1. 38As a member, I want changes made in another tab (as another identity) to appear in my open Board, so that collaboration is visible.
  2. 39As a member, I want the Board to refresh quietly without flashing the whole screen, so that updates are unobtrusive.
  3. 40As a member whose Card was deleted by someone else, I want it removed from my view, so that I don't act on a stale Card.
  4. 41As a member who is removed from a Board while viewing it, I want to be ejected with a message on the next sync, so that I understand my access changed.

Robustness toggles (dev/demo) 1

  1. 42As a developer demoing the app, I want a toggle that forces mock-backend failures, so that I can demonstrate optimistic rollback on demand.

🏗️Implementation Decisions

Scope & architecture (locked — Approve PRD §1)

  • Build the React frontend only; back it with a mock backend (localStorage). No Node/PostgreSQL this round.
  • Layering: UI components → Zustand store (optimistic, persisted) → API client (PRD §3.7 method shape) → mockBackend.
  • The API client is a deliberately shallow seam (the swap point). Methods getBoards, getBoard, createBoard, patchBoard, deleteBoard, addMember, removeMember, createColumn, patchColumn, deleteColumn, createCard, patchCard, moveCard, deleteCard, createLabel, patchLabel, deleteLabel, attachLabel, detachLabel match PRD §3.7 so a real fetch() backend drops in without changing callers.

Modules

DEEPdomain/ordering — pure fractional-float ordering: positionBetween(prevPos, nextPos), needsRebalance(positions), rebalance(items). Moving one item computes a new position from neighbours and writes a single record; rebalance runs when the float gap collapses.
DEEPdomain/validation — pure validators encoding §3.10: board/column name non-empty ≤100; card title non-empty ≤255; description ≤~5000; label color valid hex; invite email must match an existing seeded user.
DEEPapi/mockBackend — the "server". Canonical state, mutations, authorization (membership guard every op; owner check for board delete / member management), cascade deletes (board→columns/cards/labels/members; column→cards; label→card_labels only), latency + forced failures, persist + broadcast across tabs. Async, REST-shaped.
SHALLOWstore/useBoardStore (Zustand + persist) — orchestrates optimistic updates: snapshot slice, apply, call client, restore + surface error on rejection.
SHALLOWapi/client — thin adapter delegating to mockBackend.
SHALLOWReact components / routes — verified via manual acceptance flows.

Other decisions

  • Identity — User-switcher over seeded users; current identity persisted in localStorage; used as "current user" for all authz in mockBackend. No real password hashing/JWT.
  • Data model — PRD §3.6 logical schema (users, boards, board_members, columns, cards, labels, card_labels), UUID ids, position floats. Stored as JSON in localStorage but shaped identically so a real DB maps 1:1.
  • Card move contract — single moveCard(cardId, { columnId, position }) covers cross-Column move + in-Column reorder; mockBackend always assigns a final authoritative position (no collisions, PRD §3.9).
  • Sync — usePolling reconciles the Board snapshot on a fallback interval and on cross-tab storage events (last-write-wins); stands in for the PRD's ~10s polling.
  • Card detail — right-hand side-panel, Board visible behind (PRD §4.2 panel selected).
  • Tech — react-router-dom; dnd-kit (@dnd-kit/core + @dnd-kit/sortable, keyboard-capable); CSS Modules; uuid; seed "Project Phoenix" board (mockup §4.7).

🧪Testing Decisions

เกณฑ์เทสต์ที่ดี: assert external behavior ผ่าน public interface ของโมดูล ไม่ใช่ internal representation สำหรับ domain/ordering: ให้ position ที่คืนมาเรียงถูกตำแหน่งเทียบ siblings — assert ผลการเรียงลำดับ ไม่ใช่ค่า float เป๊ะๆ

Modules to be unit-tested (developer selected): domain/ordering only

  • Insert at head (no previous), tail (no next), and middle (between two) — ผลต้องเรียงเข้า slot ที่ตั้งใจ
  • rebalance — เมื่อ insert ถี่จน gap ตัน ต้องกระจาย position ใหม่โดยคงลำดับเดิม
  • needsRebalance — คืน true เมื่อ precision ระหว่าง position ติดกันหมด
ไม่เทสต์รอบนี้ (ตามที่ developer เลือก): domain/validation, mockBackend authz, store rollback — ครอบด้วย manual acceptance flow แทน (ยังเป็นเป้าหมายเทสต์ในอนาคตได้ เพราะ interface ออกแบบมาให้เทสต์ได้)

Prior art: src/App.test.js เดิมใช้ @testing-library/react + Jest (CRA default). เทสต์ domain/ordering ไม่ต้องใช้ DOM — Jest unit test ล้วน รันผ่าน npm test

🚫Out of Scope

  • Real backend (Node.js/Express/Fastify), PostgreSQL, real JWT/bcrypt — เลื่อนไป phase ถัดไป; API client seam คือจุดต่อ
  • Real multi-user/multi-device collaboration, WebSocket live sync, presence/cursors
  • Fine-grained roles เกิน owner/member (viewer/editor/admin)
  • Sub-checklists, large file attachments, email/push notifications, automation rules, swimlanes, cross-board advanced search, mobile native app, analytics dashboard
  • เชิญ user ที่ยังไม่ seed/register
  • Unit tests สำหรับ validation, mockBackend authz, store rollback (รอบนี้ครอบด้วย manual เท่านั้น)
